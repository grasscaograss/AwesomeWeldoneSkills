# 如何归纳 Spec

## 读取待处理提交

```bash
# 列出源分支独有的提交（从旧到新）
git log --reverse --oneline <source> ^<target>
```

从进度文件的"待处理提交"列表中取下一批未处理的提交。

## 归纳标准

**属于同一个 spec 的特征：**
- 提交消息中有相同的功能关键词（如 `feat(scan)`, `fix(scan)`）
- 涉及的文件属于同一模块/目录
- 连续的提交且彼此有依赖（后一个修复了前一个引入的问题）
- 同一功能的拆分提交（WIP, part1/part2, 补丁等）

**应该拆成不同 spec 的情况：**
- 涉及完全不同的业务模块
- 一个是 bug fix，一个是新功能
- 文件无交集且语义无关联

## 归纳步骤

1. 读提交消息，找聚类关键词
2. `git show --stat <hash>` 确认涉及的文件范围
3. 对不确定的提交，读 diff：`git show <hash>`
4. 写出意图描述：**这组提交实现了什么？解决了什么问题？**

## 判断实现方式

| 情况 | 推荐方式 |
|---|---|
| 改动 ≤ 3 个文件，逻辑清晰 | 直接迁移 |
| 简单 bug fix 或配置调整 | 直接迁移 |
| 改动涉及多个模块，意图复杂 | opsx:new |
| 需要在目标分支重新设计实现方案 | opsx:new |
| 应用过程中发现比预期复杂 | 停下来改用 opsx:new |

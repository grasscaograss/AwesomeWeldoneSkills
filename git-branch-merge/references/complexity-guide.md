# 合并复杂度评估指南

根据诊断结果，判断整体策略。

## 复杂度矩阵

| 独有提交数 | 文件差异数 | 推荐策略 |
|---|---|---|
| < 20 个 | < 50 个文件 | 直接 `git merge`，手动解冲突 |
| 20-100 个 | 50-200 个文件 | **逐 spec 迁移**（本 skill 默认流程） |
| > 100 个 | > 200 个文件 | 逐 spec 迁移，优先迁移 bug fix 和基础设施 |
| 任意 | 有大量重命名/删除 | 先通过 `git diff --name-status` 摸清文件变动，再开始迁移 |

## 分支角色判断

- **目标分支（target）**：需要接收改动的分支（通常是主力开发分支，提交数更多）
- **源分支（source）**：需要被迁移的分支（提交数较少，改动要引入到 target）

## 大量删除文件的处理

当源分支有大量文件删除时：

```bash
# 查看源分支相对于目标分支删除了哪些文件
git diff --name-status <target> <source> | grep "^D" | head -30
```

- 目标分支仍依赖这些文件 → 迁移时跳过删除，记入进度文件备注
- 文件已被重构/替代 → 评估目标分支是否需要同步删除相关引用
